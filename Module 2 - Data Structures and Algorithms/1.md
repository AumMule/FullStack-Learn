### 🔍 Exercise 2: E-commerce Platform Search Function
*   **Scenario:** Implementing optimized search functionality (Linear Search vs. Binary Search) for product searching.
*   **Big O Notation:** Describes the limiting execution behavior of an algorithm as the input size ($N$) grows, providing a mathematical upper bound on runtime or memory usage.
*   **Files:**
    *   [Product.java](file:///c:/Users/ommul/OneDrive/Desktop/cognizant/Module%202%20-%20Data%20Structures%20and%20Algorithms/EcommercePlatformSearch/Product.java)
    *   [SearchAlgorithms.java](file:///c:/Users/ommul/OneDrive/Desktop/cognizant/Module%202%20-%20Data%20Structures%20and%20Algorithms/EcommercePlatformSearch/SearchAlgorithms.java)
    *   [SearchTest.java](file:///c:/Users/ommul/OneDrive/Desktop/cognizant/Module%202%20-%20Data%20Structures%20and%20Algorithms/EcommercePlatformSearch/SearchTest.java)

#### 📊 Complexity Comparison:
| Search Method | Best Case | Average Case | Worst Case | Prerequisite |
| :--- | :--- | :--- | :--- | :--- |
| **Linear Search** | $O(1)$ | $O(N)$ | $O(N)$ | None (Works on unsorted arrays) |
| **Binary Search** | $O(1)$ | $O(\log N)$ | $O(\log N)$ | Array **MUST** be sorted |

> [!TIP]
> **Which one to choose?** For read-heavy catalogs (where searching is highly frequent but edits are rare), maintaining a sorted list and using **Binary Search** is highly superior. For dynamic, small catalogs where sorting cost dominates, Linear Search may suffice.

#### 📊 Performance Matrix:
| Algorithm | Best Case | Average Case | Worst Case | Space Complexity |
| :--- | :--- | :--- | :--- | :--- |
| **Bubble Sort** | $O(N)$ (if optimized) | $O(N^2)$ | $O(N^2)$ | $O(1)$ (In-place) |
| **Quick Sort** | $O(N \log N)$ | $O(N \log N)$ | $O(N^2)$ | $O(\log N)$ (Recursive Stack) |

> [!NOTE]
> **Why is Quick Sort Preferred?** Even though its worst-case is $O(N^2)$ (rare in practice if randomized pivot is used), Quick Sort has very small constant factor overheads, is highly cache-friendly, and operates in-place.

*   **Linked Lists Concept:** Singly Linked Lists contain nodes that reference only the next element, while Doubly Linked Lists support backward traversal by keeping references to both the previous and next nodes.
*   **Files:**
    *   [Task.java](file:///c:/Users/ommul/OneDrive/Desktop/cognizant/Module%202%20-%20Data%20Structures%20and%20Algorithms/TaskManagementSystem/Task.java)

#### 📊 Operations Complexity (Singly Linked List):
| Operation | Complexity | Rationale |
| :--- | :--- | :--- |
| **Add** | $O(1)$ / $O(N)$ | $O(1)$ if added to the head (or using a tail pointer). $O(N)$ if traversing from head to locate the end. |
| **Search** | $O(N)$ | Must traverse linearly from the head node. |
| **Traverse** | $O(N)$ | Sequential iteration through all pointers. |
| **Delete** | $O(N)$ | Needs linear scan to locate the task node, followed by an $O(1)$ pointer update. |

### 📈 Exercise 7: Financial Forecasting
*   **Scenario:** Predicting compounding future returns recursively.
*   **Recursion:** A mechanism where a method resolves a complex problem by calling a smaller sub-instance of itself until it reaches a terminal base case.
*   **Files:**
    *   [Forecasting.java](file:///c:/Users/ommul/OneDrive/Desktop/cognizant/Module%202%20-%20Data%20Structures%20and%20Algorithms/FinancialForecasting/Forecasting.java)
    *   [ForecastingTest.java](file:///c:/Users/ommul/OneDrive/Desktop/cognizant/Module%202%20-%20Data%20Structures%20and%20Algorithms/FinancialForecasting/ForecastingTest.java)

#### 📊 Recursion Analysis:
*   **Time Complexity:** $O(N)$ where $N$ is the number of periods/years, as it makes exactly $N$ recursive call divisions.
*   **Space Complexity:** $O(N)$ due to call stack frames created for each call.
*   **Optimization Advice:** To avoid stack overflow errors on extremely large values of $N$, convert the recursive call into an **Iterative Loop**, reducing space complexity from $O(N)$ to $O(1)$.
