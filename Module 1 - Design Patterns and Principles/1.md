# SOLID Principles

SOLID is an acronym for five design principles of object-oriented programming that help make software designs more understandable, flexible, and maintainable.

---

## The 5 Principles

```
S - Single Responsibility Principle
O - Open/Closed Principle
L - Liskov Substitution Principle
I - Interface Segregation Principle
D - Dependency Inversion Principle
```

---

### 1. Single Responsibility Principle (SRP)
> **"A class should have only one reason to change."**

A class should focus on a single job or concern. Keeping classes focused makes testing easier and reduces coupling.

#### ❌ The Bad Way
A `Book` class that handles both book properties and printing/formatting logic:
```java
public class Book {
    private String name;
    private String author;
    private String text;

    public String replaceWordInText(String word, String replacementWord) {
        return text.replaceAll(word, replacementWord);
    }
    
    // Violates SRP: Printing logic inside Book
    void printTextToConsole() {
        System.out.println(text);
    }
}
```

#### ✅ Refactored Way
Separate the data/logic of the book from how it is printed:
```java
public class Book {
    private String name;
    private String author;
    private String text;

    public String replaceWordInText(String word, String replacementWord) {
        return text.replaceAll(word, replacementWord);
    }
}

// Dedicated class for printing concern
public class BookPrinter {
    void printTextToConsole(String text) {
        System.out.println(text);
    }

    void printTextToAnotherMedium(String text) {
        // Send email, log to file, etc.
    }
}
```

### 2. Open/Closed Principle (OCP)
> **"Classes should be open for extension, but closed for modification."**

You should be able to add new functionality without modifying existing, tested code (preventing regression bugs).

#### ❌ The Bad Way
Modifying a completed and deployed class directly:
```java
// Original class
public class Guitar {
    private String make;
    private String model;
    private int volume;
}

// Modifying the class later to add custom attributes (Risky!)
public class Guitar {
    private String make;
    private String model;
    private int volume;
    private String flameColor; // Added directly to the original class
}
```
#### ✅ Refactored Way
Extend the class to add features:
```java
public class Guitar {
    private String make;
    private String model;
    private int volume;
}

// Open for extension: Inheriting to add flame pattern
public class SuperCoolGuitarWithFlames extends Guitar {
    private String flameColor;
}
```

### 3. Liskov Substitution Principle (LSP)
> **"If S is a subtype of T, then objects of type T may be replaced with objects of type S without altering the correctness of the program."**

Subclasses should behave in a way that doesn't break the expectations set by the superclass.

#### ❌ The Bad Way
Implementing an interface with a subclass that throws errors for basic interface methods:
```java
public interface Car {
    void turnOnEngine();
    void accelerate();
}

public class MotorCar implements Car {
    private Engine engine;
    public void turnOnEngine() { engine.on(); }
    public void accelerate() { engine.powerOn(1000); }
}

// Violates LSP: ElectricCar throws exception because it has no engine
public class ElectricCar implements Car {
    public void turnOnEngine() {
        throw new AssertionError("I don't have an engine!");
    }
    public void accelerate() {
        // Acceleration logic...
    }
}
```

#### ✅ Refactored Way
Rework the abstractions so that classes are not forced to implement behavior they don't support (e.g. separating the concept of a `Car` from having an `Engine`).

### 4. Interface Segregation Principle (ISP)
> **"Clients should not be forced to depend on methods they do not use."**

Split large, monolithic interfaces into smaller, more specific ones so that implementing classes only care about methods relevant to them.

#### ❌ The Bad Way
A monolithic interface that forces classes to implement methods they don't want or need:
```java
public interface BearKeeper {
    void washTheBear();
    void feedTheBear();
    void petTheBear(); // Dangerous and unnecessary for standard caretakers
}
```

#### ✅ Refactored Way
Split the large interface into dedicated roles:
```java
public interface BearCleaner {
    void washTheBear();
}

public interface BearFeeder {
    void feedTheBear();
}

public interface BearPetter {
    void petTheBear();
}

// Now caretakers only implement what they safely can do:
public class BearCarer implements BearCleaner, BearFeeder {
    public void washTheBear() { /* wash logic */ }
    public void feedTheBear() { /* feed logic */ }
}

public class CrazyPerson implements BearPetter {
    public void petTheBear() { /* pet logic */ }
}
```

### 5. Dependency Inversion Principle (DIP)
> **"Depend on abstractions, not on concretions."**

High-level modules should not depend on low-level modules; both should depend on abstractions (interfaces).

#### ❌ The Bad Way
Tightly coupling a machine with concrete parts:
```java
public class Windows98Machine {
    private final StandardKeyboard keyboard;
    private final Monitor monitor;

    // Violates DIP: Tightly coupled to specific classes StandardKeyboard and Monitor
    public Windows98Machine() {
        this.monitor = new Monitor(); 
        this.keyboard = new StandardKeyboard(); 
    }
}
```

#### ✅ Refactored Way
Use an interface abstraction and inject it via constructor (Dependency Injection):
```java
public interface Keyboard { }

public class StandardKeyboard implements Keyboard { }

public class Windows98Machine {
    private final Keyboard keyboard;
    private final Monitor monitor;

    // Loosely coupled: Keyboard is an abstraction
    public Windows98Machine(Keyboard keyboard, Monitor monitor) {
        this.keyboard = keyboard;
        this.monitor = monitor;
    }
}
```
