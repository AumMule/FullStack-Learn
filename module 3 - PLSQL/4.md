## 📦 14. Packages

Packages act as containers that group logically related variables, constants, subprograms, cursors, and exceptions. Similar to a Java class definition, packages are divided into two parts:

### A. Specification (Interface / Header)
Declares all public components accessible outside the package.
#### Syntax
```sql
CREATE [OR REPLACE] PACKAGE package_name IS
   -- Variable, constant, cursor declarations
   -- Procedure & Function signatures
END package_name;
/
```
#### Example
```sql
CREATE OR REPLACE PACKAGE math_pkg IS
   FUNCTION add_nums(a NUMBER, b NUMBER) RETURN NUMBER;
END math_pkg;
/
```

### B. Body (Implementation)
Implements all procedures and functions declared in the specification, as well as private variables.
#### Syntax
```sql
CREATE [OR REPLACE] PACKAGE BODY package_name IS
   -- Procedure & Function implementations
END package_name;
/
```
#### Example
```sql
CREATE OR REPLACE PACKAGE BODY math_pkg IS
   FUNCTION add_nums(a NUMBER, b NUMBER) RETURN NUMBER IS
   BEGIN
      RETURN a + b;
   END add_nums;
END math_pkg;
/
```

## ⚡ 15. Triggers

Triggers are stored subprograms executed automatically in response to database events (such as inserting, updating, or deleting rows).

### A. Core Types
*   **`BEFORE`**: Fires before the operation. Used for validation, checks, or modifying input values.
*   **`AFTER`**: Fires after the operation. Used for logging, auditing, or cascading updates.

### B. Special Variables
Inside row-level triggers (`FOR EACH ROW`), you have access to two pseudo-record variables:
*   **`:OLD.column_name`**: References the column value *before* the modification.
*   **`:NEW.column_name`**: References the column value *after* the modification.

| Operation | `:OLD` | `:NEW` |
| :--- | :--- | :--- |
| **`INSERT`** | All fields are `NULL` | Contains values being inserted |
| **`UPDATE`** | Contains original values | Contains new update values |
| **`DELETE`** | Contains values before delete | All fields are `NULL` |

### Syntax & Example
```sql
CREATE OR REPLACE TRIGGER audit_salary_update
BEFORE UPDATE ON employees
FOR EACH ROW
BEGIN
   -- Log old and new salary details
   IF :OLD.salary != :NEW.salary THEN
      DBMS_OUTPUT.PUT_LINE('Employee ID: ' || :OLD.employee_id);
      DBMS_OUTPUT.PUT_LINE('Old Salary : ' || :OLD.salary);
      DBMS_OUTPUT.PUT_LINE('New Salary : ' || :NEW.salary);
   END IF;
END;
/
```

## 🎓 16. Exam Readiness Summary

Quick-reference mapping of PL/SQL features to support rapid recall:

| Feature | Primary Goal | Java Equivalent | Cheat Sheet Syntax / Tip |
| :--- | :--- | :--- | :--- |
| **Block** | Logical grouping | Code scope (`{ ... }`) | `DECLARE / BEGIN / EXCEPTION / END;` |
| **`%TYPE`** | Safe variable binding | Variable inheritance | `variable table.column%TYPE;` |
| **`%ROWTYPE`** | Match table records | Class / Data Structure | `record table%ROWTYPE;` |
| **`SELECT INTO`** | Populate variables | Single element lookup | Must return **exactly 1 row** (No more, no less). |
| **`ELSIF`** | Conditional branch | `else if` | Spelling: **`ELSIF`** (no E in middle). |
| **`EXIT WHEN`** | Break out of loop | `break` | `EXIT WHEN condition;` inside `LOOP ... END LOOP;` |
| **Cursor** | Multiline collection | `Iterator` | `FOR emp IN (SELECT ...) LOOP` is cleanest. |
| **Procedure** | Operation command | `void` method | `CREATE PROCEDURE p IS BEGIN ... END;` |
| **Function** | Computation | Returning method | `CREATE FUNCTION f RETURN T IS BEGIN RETURN x; END;` |
| **`OUT` parameter**| Value passback | By-reference output | Re-assign inside subprogram to pass back to caller. |
| **Package** | Logic organization | Class namespaces | Split into **Specification** and **Body**. |
| **Trigger** | Automated event hook | Event listener | Row-level triggers use `:OLD` and `:NEW`. |
